Credit Card Fraud detection using Python

A study on stacking models to achieve better prediction results.

Summary

As estimated, companies lose around 5% of their revenues to fraud. The credit card fraud dataset is one of the most popular datasets in the data science community, leading to a huge number of predictor implementations. This model uses a stacked logistic regressor and an XGboost model to predict the fraudulent transactions with 99.90% accuracy and 0.01% type II errors. While it may be good, the fact that such errors happen in systems where thousands of transactions are processed each second, the value needs to be further minimised.

Model performance analysis

The logistic regression model demonstrates a fairly good performance with:
- 99.90% accuracy
- 36 Type II (false negative) errors per 76,000 test input rows
- 39 Type I (false positive) errors per 76,000 test input rows
- 81 correctly classified fraud transactions.
Per the numbers above, we can see that 0.04% of all transactions are misclassified as non-fraud.

The XGBoost model has proven to be a better-suiting model for this dataset, with:
- 99.95% accuracy
- 8 Type II (false negative) errors per 76,000 input test rows
- 30 Type I (false positive) errors per 76,000 input test rows
- 90 correctly classified fraud transactions.
While the percentage of missed fraud transactions has improved (up to 0.01%), 8 errors per 76,000 transactions are still a big number.

The stack model, composed of logistic regression and XGBoost, has performed the best, with: 99.90% accuracy
- 1 Type II (false negative) error per 76,000 input test rows
- 46 Type I (false positive) errors per 76,000 input test rows
- 79 correctly classified fraud transaction.
The reason behind the bump in type I error count is not clear to me; I suppose both models in stack need fine tuning. However, 1 misclassified fraud transaction per 76,000 transactions is a drastic improvement compared to the solely used models, even though it is still a big issue. I assume more evaluation data would lead to a small bump in type II errors.
